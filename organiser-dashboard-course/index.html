How the Organiser Dashboard Works
01

Welcome to the Dashboard

One place instead of six tabs

💬
The problem

Community organisations run their lives across Discourse forums, Matrix chat, a blog, YouTube, and Instagram — and everyone is checking 5–6 different tabs constantly.

🗂️
The solution

The Organiser Dashboard pulls all of those sources into one live-updating page. No switching tabs. No manual refreshes. Everything in one place.

🔓
The philosophy

Every service it talks to — Discourse, Matrix — is open-source and self-hostable. No vendor lock-in. Your data stays with you.

Eight panels, three columns

When you open the dashboard you see a three-column grid. Each "panel" is a live window into one data source. You can toggle any panel on or off — and your preferences are saved in your browser.

📌

Left Column

Announcements, Chat (Matrix), Instagram

🗣️

Centre Column

Discussions — the main Discourse forum feed

📚

Right Column

Videos, Articles (RSS), Meetings, Key Docs

💡
Why three columns?

On desktop you see all three side-by-side. On mobile they stack. The CSS order property controls which column appears first when stacked — by default, actionable content (Announcements, Chat) comes before the forum on mobile.

What happens in the first second

Imagine you type the dashboard URL in your browser and hit Enter. Here is the exact chain of events — from your keystroke to the page appearing fully populated with data.

1
Your browser sends a request to Netlify

The URL resolves to a Netlify serverless function running the Nuxt server.

2
The server fetches all data in parallel

The Nitro server calls Discourse, Matrix, YouTube, etc. simultaneously — not one after the other.

3
A complete HTML page arrives in your browser

The server renders the full HTML — panels already populated — and sends it all at once. This is called SSR.

4
The browser opens a live channel

After the page loads, JavaScript silently opens a SSE connection to the server. From now on, the server pushes fresh data automatically.

The Cast of Characters

Who does what — and where to find them in the code

Every app is a team of specialists. Understanding who's responsible for what helps you give better instructions to AI — instead of saying "put the data somewhere", you can say "put this logic in the Nitro API route, not the Vue component."

🎭
Think of it like a newsroom

The Vue components are the journalists who write the stories you read. The Nitro server is the editor who gathers the raw information. The Discourse/Matrix/YouTube services are the wire feeds — sources the newsroom monitors constantly.

The front end: Vue + Nuxt

The front end is what runs in your browser. It's built with Vue, wrapped in a framework called Nuxt.

app/ Everything the browser sees and runs
components/panels/ One .vue file per panel — DiscussionsPanel, MatrixPanel, etc.
components/ui/ Shared widgets — PanelWrapper, ThemeSwitcher, UserPreferences
composables/ Reusable logic — useSSE, usePanelVisibility, useColorTheme
pages/index.vue The dashboard grid — arranges the three columns and panels
layouts/default.vue The sidebar and header that wrap every page
💡
Key insight: composables

A composable is a function that bundles related logic together. useSSE bundles all the SSE connection logic. usePanelVisibility bundles all the show/hide logic. When you tell AI to add a feature, knowing "put this in a composable" keeps the codebase organised.

The back end: Nitro API routes

The back end is built with Nitro. Each panel has its own dedicated API route.

server/ Everything that runs on the server, never exposed to users
api/ One file per endpoint — the file name IS the URL
discourse/latest.get.ts GET /api/discourse/latest — forum topics
pinboard/index.get.ts GET /api/pinboard — announcements
matrix/index.get.ts GET /api/matrix — chat messages
events/stream.get.ts GET /api/events/stream — the live SSE connection
utils/ Shared helpers — cache.ts, discourse.ts, eventBus.ts

The external services

The dashboard doesn't store most of its own data — it reads from services your organisation already uses. These are configured via environment variables, so the same code can power any organisation's dashboard with no code changes.

03

Always Fresh

The live update engine — SSE, caching, and the event bus

Most dashboards get "live" data the lazy way: they ask the server for new data every 30 seconds. This is called polling, and it's expensive and slow.

📻
The radio station model

This dashboard works like a radio station instead. The server broadcasts updates when it has something new. The browser just listens. No constant nagging. The moment data changes, every connected browser gets it simultaneously — without asking.

This is achieved with three interlocking systems: the SSE stream, the in-memory cache, and the event bus. Let's look at each one.

System 1: The in-memory cache

Every API route has its own cache. When Nitro fetches data from Discourse, it stores the result in memory with a timer. If another request comes in before the timer expires, it returns the cached copy instead of calling Discourse again.

CODE — server/utils/cache.ts 

export function createCache<T>(ttlMs: number) {
  let data: T | null = null
  let timestamp: number | null = null
  return {
    isFresh() {
      return data !== null &&
        (Date.now() - timestamp!) < ttlMs
    },
    get() { return data },
    set(val: T) {
      data = val
      timestamp = Date.now()
    }
  }
}
PLAIN ENGLISH

Create a cache that holds data of type T for ttlMs milliseconds.

The data holder — starts empty.

The timestamp — when we last filled it.

Return a small toolkit with three tools:

isFresh: Is the data still good? Check if we have data…

…and if it's younger than the expiry time.

get: Just hand back whatever's stored.

set: Store new data and record the current time as the fresh timestamp.

5 min Discourse topics, Pinboard, Meetings — active conversation data
2 min Matrix chat — faster refresh because chat moves quickly
30 min Blog articles, YouTube videos, Instagram — changes rarely
No cache Pinboard writes — always fresh, because you just changed it

System 2: The event bus

When a Nitro API route fetches fresh data, it needs to tell the SSE stream so it can broadcast to all connected browsers. But how does one part of the server talk to another? Through an event bus.

CODE — server/utils/eventBus.ts 

import { EventEmitter } from 'node:events'

const eventBus = new EventEmitter()
eventBus.setMaxListeners(50)

export default eventBus
PLAIN ENGLISH

Use Node.js's built-in message board system.

Create the shared message board.

Allow up to 50 listeners — prevents false "memory leak" warnings.

Share this one message board with the whole server.

Here's how it's used in practice. The Discourse API route fetches new topics, then posts to the event bus:

CODE — server/api/discourse/latest.get.ts 

const payload = {
  source: 'live',
  topics: transformed
}
eventBus.emit('discourse', payload)
return payload
PLAIN ENGLISH

Package up the result.

Label it as real live data (not mock or cached).

Include the freshly fetched topics.

Post the message "discourse" to the shared board with this payload.

Also return the data to whoever called this function directly.

System 3: The SSE stream

The SSE stream (server/api/events/stream.get.ts) is the hub. It listens on the event bus and broadcasts to every connected browser. It also starts background timers that trigger periodic refreshes automatically.

CODE — server/api/events/stream.get.ts (excerpt) 

// Wire bus events to broadcast
eventBus.on('discourse', (data) =>
  broadcast('discourse', data)
)

// Background refresh timers
setInterval(
  () => fetchLatestTopics(),
  5 * 60 * 1000 // 5 min
)
PLAIN ENGLISH

Subscribe to the "discourse" message on the event bus.

Whenever a "discourse" message arrives from anywhere in the server…

…immediately send it to every connected browser.

Set up a repeating timer.

Every 5 minutes, go fetch the latest Discourse topics.

5 × 60 × 1000 = 5 minutes in milliseconds.

💡
Why this design is clever

The timer runs on the server, not in each browser. So even if 200 members have the dashboard open, Discourse only gets asked for new data once every 5 minutes — not 200 times. Everyone's browser gets the same broadcast simultaneously. This is called a fan-out pattern.

04

Discourse as the Backbone

Why one forum becomes the source of truth for everything

Most dashboards keep data in a separate database. This one doesn't. Instead, Discourse serves as the single source of truth for almost all written content in the dashboard.

🗣️
Discussions panel

Reads from Discourse categories you configure — shows latest topics with excerpts, reply counts, and timestamps.

📌
Announcements (Pinboard) panel

Reads from a special Discourse category — AND can write back to it, creating real Discourse topics when you post an announcement.

📅
Meetings panel

Reads from a separate Discourse category designated for meeting notes and agendas.

📄
Key Docs panel

Reads from yet another Discourse category — for reference documents and policies.

Why not just use a database?

🏛️
Discourse is a database — with superpowers

A traditional database is just storage. Discourse gives you storage plus moderation tools, search, notifications, user accounts, edit history, and an existing community already living there. Why build all that from scratch?

✓
Everything is auditable

Every announcement, meeting note, and key doc has a full edit history in Discourse. Who wrote it, when, and what changed.

✓
Moderation is built in

Discourse has user roles, category permissions, and spam protection. The dashboard inherits all of that for free.

✓
Self-hosted and open-source

Unlike Slack or Notion, Discourse can run on your own server. Your organisation owns all its data.

✓
The dashboard degrades gracefully

If Discourse isn't configured, every panel falls back to realistic mock data. Nothing breaks — it just shows example content.

How Nitro talks to Discourse

Every API route that needs Discourse data goes through a shared helper function in server/utils/discourse.ts. This keeps credentials in one place and makes every API call consistent.

CODE — server/utils/discourse.ts 

export async function discourseRequest<T>(
  path: string,
  method = 'GET'
) {
  const config = useRuntimeConfig()
  const url = new URL(path, config.discourseUrl)
  const headers = {
    'Api-Key': config.discourseApiKey,
    'Api-Username': config.discourseApiUsername
  }
  return await $fetch<T>(url.toString(), {
    method, headers, timeout: 8000
  })
}
PLAIN ENGLISH

A reusable function that makes one HTTP request to Discourse.

You provide a path like /c/5/l/latest.json…

…and optionally the method (GET by default).

Pull the server-side config (env vars, invisible to the browser).

Build the full URL by combining the Discourse base URL + the path.

Prepare the authentication headers — the API key and username.

These prove to Discourse that this request is from an authorised source.

Make the HTTP request and wait for the response.

Give up after 8 seconds if Discourse doesn't respond.

Make It Your Own

Panel visibility, colour themes, localStorage, and zero-code configuration

One of the cleverest things about this dashboard is how much users can customise without touching code — and how much you can configure without touching code either. This module explains the three layers of customisation.

👤

User preferences

Panel visibility, colour theme, text size — saved per device in localStorage

⚙️

Operator config

Org name, colours, Discourse URL, API keys — set via environment variables

📢

Announcements

Priority banners — edited by changing a JSON file on GitHub, no code needed

localStorage: your browser's sticky note pad

localStorage is how the dashboard remembers your preferences. No account required. No server round-trip. Everything stays on your device.

panel-visibility-v1 Which of the 8 panels you have switched on or off
dash-color-theme Your chosen colour theme: coffee, forest, tokyo, or dyslexic
dash-font-size Your text size preference: 16px, 18px, or 20px
announcement-dismissed-{hash} Whether you dismissed a particular announcement (keyed by content hash)
💡
The version number trick

Notice panel-visibility-v1 has a version number. When the developer changes the default panels, they bump it to panel-visibility-v2. Existing users' saved preferences are ignored — they get a clean start with the new defaults. This is a standard pattern for breaking changes in user-facing storage.

How panel toggling actually works

When you click a panel toggle in the sidebar, here's what happens. The magic is v-if — panels that are off are completely removed from the browser, not just hidden.

CODE — app/composables/usePanelVisibility.ts 

function toggle(id: PanelId) {
  const currentlyOn = visibility.value[id]
  if (currentlyOn && !canToggleOff(id))
    return

  visibility.value = {
    ...visibility.value,
    [id]: !currentlyOn
  }
  saveToStorage(visibility.value)
}
PLAIN ENGLISH

Function to toggle one panel on or off.

Check if the panel is currently on.

If it's on, and turning it off would leave fewer than 5 panels…

…stop. The minimum is enforced, no error thrown.

Otherwise, create a new state object.

Copy all current on/off values (the ... spread)…

…but flip this panel's value to the opposite of what it was.

Immediately save to localStorage so it persists after a refresh.

💡
v-if vs v-show — a critical difference

v-if="false" completely removes the element from the browser — as if it never existed. No API calls fire, no SSE updates are processed. v-show="false" just hides it with CSS but the code still runs. The dashboard uses v-if deliberately — turning off a panel genuinely stops all background activity for it. This is the right call for performance.

Four colour themes — loaded on demand

The "Your Style" panel offers four colour themes. Most are just CSS — but the Dyslexic theme has a clever trick: it loads the OpenDyslexic font only when selected, rather than adding it to every page load.

CODE — app/composables/useColorTheme.ts 

function applyTheme(theme: ColorThemeName) {
  const html = document.documentElement
  if (theme === 'coffee') {
    html.removeAttribute('data-color-theme')
  } else {
    html.setAttribute(
      'data-color-theme', theme
    )
  }
  if (theme === 'dyslexic') {
    loadDyslexicFont()
  }
}
PLAIN ENGLISH

Function that switches the colour theme.

Get the root HTML element — the entire page hangs off this.

If the theme is "coffee" (the default)…

…remove any theme attribute — CSS defaults take over.

For any other theme…

…stamp a data-color-theme attribute on the HTML tag.

CSS rules like [data-color-theme="forest"] { ... } then kick in.

Special case: if Dyslexic is chosen…

…load the OpenDyslexic font from an external CDN — only now, on demand.

💡
Lazy loading — load things only when needed

Lazy loading the font means most users (99%+ won't choose the Dyslexic theme) never download it. This is a good pattern to ask AI for: "only load this resource when the user activates this feature."

The priority announcement banner

There's a special panel that bypasses all user preferences — the Priority Announcement Banner. It's for genuine emergencies. An organiser can post one by editing a single JSON file on GitHub.

ℹ️

info

Blue banner. Dismissable — user can close it. Reappears if content changes.

⚠️

warning

Amber banner. Dismissable, same as info.

🚨

urgent

Red banner. Not dismissable — stays until the operator sets enabled: false.

🔑
The dismiss hash trick

If a user dismissed an announcement, the dashboard stores a hash of its content in localStorage. If the announcement text changes, the hash changes — and the "dismissed" record no longer matches, so the banner reappears for everyone who had dismissed the previous version. Smart!

What you can do with this knowledge

You've just learned the full architecture of a production-grade real-time web application. Here's how it makes you a better AI director:

→
"Add a new panel for Mastodon"

You know: create a new API route in server/api/, a new Vue component in app/components/panels/, add it to usePanelVisibility.ts, and wire up the SSE event in index.vue.

→
"The Discourse panel is slow"

You know: check the cache TTL in discourse/latest.get.ts. Is it set too low? Is the Discourse server slow to respond? Is the cache being cleared too aggressively?

→
"Can we sync prefs across devices?"

You know: right now preferences are in localStorage (per device). To sync, you'd need to store them server-side — either in Discourse user preferences or a simple database.

→
"Why does the live dot go grey sometimes?"

You know: that's useSSEConnected() responding to an EventSource error. The browser auto-reconnects — but you could add a retry counter and alert if it fails more than N times.

03

Always Fresh

The live update engine — SSE, caching, and the event bus

Most dashboards get "live" data the lazy way: they ask the server for new data every 30 seconds. This is called polling, and it's expensive and slow.

📻
The radio station model

This dashboard works like a radio station instead. The server broadcasts updates when it has something new. The browser just listens. No constant nagging. The moment data changes, every connected browser gets it simultaneously — without asking.

This is achieved with three interlocking systems: the SSE stream, the in-memory cache, and the event bus. Let's look at each one.

System 1: The in-memory cache

Every API route has its own cache. When Nitro fetches data from Discourse, it stores the result in memory with a timer. If another request comes in before the timer expires, it returns the cached copy instead of calling Discourse again.

CODE — server/utils/cache.ts 

export function createCache<T>(ttlMs: number) {
  let data: T | null = null
  let timestamp: number | null = null
  return {
    isFresh() {
      return data !== null &&
        (Date.now() - timestamp!) < ttlMs
    },
    get() { return data },
    set(val: T) {
      data = val
      timestamp = Date.now()
    }
  }
}
PLAIN ENGLISH

Create a cache that holds data of type T for ttlMs milliseconds.

The data holder — starts empty.

The timestamp — when we last filled it.

Return a small toolkit with three tools:

isFresh: Is the data still good? Check if we have data…

…and if it's younger than the expiry time.

get: Just hand back whatever's stored.

set: Store new data and record the current time as the fresh timestamp.

5 min Discourse topics, Pinboard, Meetings — active conversation data
2 min Matrix chat — faster refresh because chat moves quickly
30 min Blog articles, YouTube videos, Instagram — changes rarely
No cache Pinboard writes — always fresh, because you just changed it

System 2: The event bus

When a Nitro API route fetches fresh data, it needs to tell the SSE stream so it can broadcast to all connected browsers. But how does one part of the server talk to another? Through an event bus.

CODE — server/utils/eventBus.ts 

import { EventEmitter } from 'node:events'

const eventBus = new EventEmitter()
eventBus.setMaxListeners(50)

export default eventBus
PLAIN ENGLISH

Use Node.js's built-in message board system.

Create the shared message board.

Allow up to 50 listeners — prevents false "memory leak" warnings.

Share this one message board with the whole server.

Here's how it's used in practice. The Discourse API route fetches new topics, then posts to the event bus:

CODE — server/api/discourse/latest.get.ts 

const payload = {
  source: 'live',
  topics: transformed
}
eventBus.emit('discourse', payload)
return payload
PLAIN ENGLISH

Package up the result.

Label it as real live data (not mock or cached).

Include the freshly fetched topics.

Post the message "discourse" to the shared board with this payload.

Also return the data to whoever called this function directly.

System 3: The SSE stream

The SSE stream (server/api/events/stream.get.ts) is the hub. It listens on the event bus and broadcasts to every connected browser. It also starts background timers that trigger periodic refreshes automatically.

CODE — server/api/events/stream.get.ts (excerpt) 

// Wire bus events to broadcast
eventBus.on('discourse', (data) =>
  broadcast('discourse', data)
)

// Background refresh timers
setInterval(
  () => fetchLatestTopics(),
  5 * 60 * 1000 // 5 min
)
PLAIN ENGLISH

Subscribe to the "discourse" message on the event bus.

Whenever a "discourse" message arrives from anywhere in the server…

…immediately send it to every connected browser.

Set up a repeating timer.

Every 5 minutes, go fetch the latest Discourse topics.

5 × 60 × 1000 = 5 minutes in milliseconds.

💡
Why this design is clever

The timer runs on the server, not in each browser. So even if 200 members have the dashboard open, Discourse only gets asked for new data once every 5 minutes — not 200 times. Everyone's browser gets the same broadcast simultaneously. This is called a fan-out pattern.

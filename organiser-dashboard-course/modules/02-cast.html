The Cast of Characters

Who does what — and where to find them in the code

Every app is a team of specialists. Understanding who's responsible for what helps you give better instructions to AI — instead of saying "put the data somewhere", you can say "put this logic in the Nitro API route, not the Vue component."

🎭
Think of it like a newsroom

The Vue components are the journalists who write the stories you read. The Nitro server is the editor who gathers the raw information. The Discourse/Matrix/YouTube services are the wire feeds — sources the newsroom monitors constantly.

The front end: Vue + Nuxt

The front end is what runs in your browser. It's built with Vue, wrapped in a framework called Nuxt.

app/ Everything the browser sees and runs
components/panels/ One .vue file per panel — DiscussionsPanel, MatrixPanel, etc.
components/ui/ Shared widgets — PanelWrapper, ThemeSwitcher, UserPreferences
composables/ Reusable logic — useSSE, usePanelVisibility, useColorTheme
pages/index.vue The dashboard grid — arranges the three columns and panels
layouts/default.vue The sidebar and header that wrap every page
💡
Key insight: composables

A composable is a function that bundles related logic together. useSSE bundles all the SSE connection logic. usePanelVisibility bundles all the show/hide logic. When you tell AI to add a feature, knowing "put this in a composable" keeps the codebase organised.

The back end: Nitro API routes

The back end is built with Nitro. Each panel has its own dedicated API route.

server/ Everything that runs on the server, never exposed to users
api/ One file per endpoint — the file name IS the URL
discourse/latest.get.ts GET /api/discourse/latest — forum topics
pinboard/index.get.ts GET /api/pinboard — announcements
matrix/index.get.ts GET /api/matrix — chat messages
events/stream.get.ts GET /api/events/stream — the live SSE connection
utils/ Shared helpers — cache.ts, discourse.ts, eventBus.ts

The external services

The dashboard doesn't store most of its own data — it reads from services your organisation already uses. These are configured via environment variables, so the same code can power any organisation's dashboard with no code changes.

🗣️
Discourse

Open-source forum platform. The dashboard treats it as the single source of truth for discussions, announcements, meetings, and key documents.

💬
Matrix

Open-source, decentralised chat. The dashboard reads recent messages and can send replies — all without leaving the page.

▶️
YouTube & Instagram

Via the YouTube Data API v3 and Behold.so (an Instagram widget service). Both fall back to mock data if not configured.

📰
RSS / Atom feeds

Any blog that publishes an RSS feed. WordPress, Ghost, Substack — just point the dashboard at the feed URL.

The panel grid in plain English

Here's the heart of pages/index.vue — the code that arranges the panels and wires them to live updates:

CODE 

const sse = useSSE()

// Wire SSE events to panels
sse.on('discourse', (data) =>
  discussionsPanel.value?.updateFromSSE(data)
)
sse.on('matrix', (data) =>
  matrixPanel.value?.updateFromSSE(data)
)

onMounted(() => { sse.connect() })
PLAIN ENGLISH

Get the live update system ready.

Tell the live update system: whenever new forum data arrives…

…hand it to the Discussions panel so it can refresh itself.

(The ?. means "only if the panel is currently visible")

Same thing for Matrix chat messages.

Once the page is fully loaded in the browser, open the live connection.
